The brief

Does a trial-readout stock actually move predictably in the pre-announcement window?

Each row of the dataset is a single announcement for a biotech trial — an interim or full readout — with daily stock prices from 180 trading days before to 180 after. The practical asks: using only information available before the announcement, can a model learn to price the move from 20 days before the event to 1 day after it?

The event-study window used to build the model — explored in full under 02 Target & Validation.

Part 2

Defining the target without letting the model see the future

The target simulates a trade: enter 20 days before the announcement, exit 1 day after.

return  =  ( day_post1 − day_pre20 )  /  day_pre20

Which days are safe to use as features?

Anything from the entry day onward — including the announcement itself — would leak information the trade doesn't have yet. Toggle the view below.

Chronological, expanding-window validation

Rows are sorted by event_date, not shuffled. The first 65% train an initial model; from there the model walks forward one announcement at a time — predict the next event, then fold it into training before predicting the one after that.

363 rows · initial training window (65%)
196 walk-forward test steps

This mirrors how the strategy would actually be deployed — every prediction uses only announcements dated before it, and the training set grows with each step rather than staying fixed.

Feature ceiling: only day_pre180 … day_pre21 are usable as raw price history. day_pre20 and day_post1 are consumed to build the target itself, and everything from day_pre19 through day_post180 — including the outcome-bearing announcement day — happens at or after the point the trade has already been entered.
Part 3

Three models, walked forward across 196 announcements

Each model was retrained at every step of the expanding window and scored against the announcement it hadn't seen yet. RMSE is on the return scale (0.10 ≈ a 10-point error in percentage return); R² is measured against always predicting the mean return.

Model 1 · Raw prices

160 raw day_pre price columns only. No dimensionality reduction, no trial metadata.

Model 2 · PLS prices

Same price history, projected onto 3 PLS components against the return target.

Model 3 · PLS + metadata

Model 2's components plus meanroll, phase, is_full and the interim-readout flags.

Reading the comparison: compressing the price history into 3 PLS components (Model 2) clearly beats using all 160 raw price columns (Model 1) — Model 1's negative R² means it does worse than just predicting the average return every time. Adding trial metadata on top (Model 3) makes RMSE slightly worse than Model 2 alone, which the notebook attributes to redundant or noisy categorical features rather than genuine signal loss.

Average feature importance

The standout signal: in Model 3, meanroll — TrialKey's own predicted probability of trial success — is the single most important non-price feature, sitting just behind the three PLS price components and ahead of every categorical trial-metadata field.
Part 4

What would move the needle next

The practical doesn't require implementing these — just identifying them from what the data and the results above expose.

01

Time-series features over PCA/PLS

Rolling means, momentum, and realised volatility on day_pre180…pre21 would likely carry more signal than dimensionality reduction on the raw daily levels, which mostly compresses autocorrelation rather than extracting predictive structure.

02

Feature selection on metadata

Since Model 3 underperformed Model 2, testing which categorical fields (phase, the interim flags) actually earn their place — versus dropping or re-encoding them — could recover the metadata's value without the noise.

03

Embeddings from terms

The dropped condition/protocol text field could be encoded (e.g. via a small language embedding) to capture disease-area or trial-design effects that phase and meanroll don't.

04

Explain the missing-price pattern

Missing stock prices aren't uniform across companies — worth checking if they align with trading-day-of-week gaps, delistings, or thin-float names, since that could itself be a liquidity signal.

05

Sector/peer-relative return

Framing the target as return relative to a biotech index or peer basket, rather than absolute return, could strip out market-wide moves unrelated to the trial itself.

06

Interaction: meanroll × phase

TrialKey's score likely means something different in Phase 1 versus Phase 3 — an interaction term could let the model use meanroll more precisely instead of as a flat linear effect.
